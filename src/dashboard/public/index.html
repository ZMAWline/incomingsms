System Architecture

                    

                        
The system runs on Cloudflare Workers with Supabase (PostgreSQL) as the database. It integrates with two external APIs:

                        
    
                            
  • Helix API — AT&T carrier operations (activate, cancel, resume, OTA refresh, IMEI change, MDN rotation, subscriber details)
    • 
                            
  • SkyLine API — Physical SMS gateway hardware control (send SMS, read/write IMEI, switch SIM, port status, reboot)
    • 
                        

                        
Workers communicate via Cloudflare Service Bindings (worker-to-worker calls). The dashboard is the main hub that proxies actions to specialized workers.

                        

                            
Dashboard (hub)
  |-- bulk-activator      Bulk SIM activation via Helix
  |-- details-finalizer   Finalize provisioning SIMs (get phone numbers)
  |-- mdn-rotator         MDN rotation, Fix SIM, IMEI operations, sim-action dispatcher
  |     |-- FIX_SIM_QUEUE   (Cloudflare Queue, batch=1, concurrency=1)
  |     |-- MDN_QUEUE        (Cloudflare Queue, batch=10, concurrency=1)
  |     |-- skyline-gateway  (service binding for hardware control)
  |-- sim-canceller       Cancel SIMs via Helix
  |-- sim-status-changer  Suspend / Restore SIMs via Helix
  |-- phone-number-sync   Sync phone numbers from Helix to DB
  |-- reseller-sync       Webhook notifications to resellers (number.online)
  |-- skyline-gateway     Gateway hardware proxy (via Supabase Edge Function bridge)
  |-- quickbooks          QBO OAuth + invoice API
  |-- ota-status-sync     Cron: sync SIM statuses from Helix (every 12h)
  |-- sms-ingest          Receives incoming SMS from physical gateways

                        

                    

                


                
                

                    
SIMs Page

                    

                        
The SIMs page displays all SIM cards in the system with filtering, sorting, and bulk actions.

                        

                            
Filters

                            
    
                                
  • Status — Filter by SIM status (active, provisioning, error, canceled, suspended)
    • 
                                
  • Reseller — Filter by assigned reseller
    • 
                                
  • Search — Free text search across ICCID, phone number, subscription ID
    • 
                                
  • Hide Cancelled — Toggle to exclude canceled SIMs from the table
    • 
                            

                        

                        

                            
Caching

                            
SIM data is cached client-side for 30 minutes. The Refresh button or changing filters force-reloads from the server. The cache timestamp is shown in the header.

                        

                        

                            
Column Data Sources

                            
Each column in the SIMs table is populated from a specific source. Understanding this helps diagnose stale or missing data.

                            

                                
                                    

                                        Column
                                        Source
                                        Written by / When
                                    

                                
                                
                                    

                                        ID
                                        sims.id
                                        Auto-incremented primary key. Set when the SIM row is first inserted (bulk-activator or manual).
                                    

                                    

                                        Gateway
                                        sims.gateway_id → gateways.code
                                        sms-ingest: updated every time an inbound SMS arrives from a port — the MAC address in the webhook is resolved to a gateway. Also updated by fix-sim / retry-activation when scanning gateways for the ICCID.
                                    

                                    

                                        ICCID
                                        sims.iccid
                                        bulk-activator: written at activation time from the CSV upload. Never changed after that (changing ICCID requires a separate Helix API call).
                                    

                                    

                                        Phone (MDN)
                                        sim_numbers.e164
(valid_to IS NULL)		
                                        mdn-rotator: inserts a new row in sim_numbers after each rotation; sets valid_to on the old row. The current MDN is always the row with valid_to = null. Initially set by bulk-activator from Helix subscriber details post-activation.
                                    

                                    

                                        Status
                                        sims.status
                                        Written by multiple workers: bulk-activator → provisioning; details-finalizer → active when Helix confirms; ota-status-sync → syncs every 12 h from Helix; mdn-rotator / dashboard actions update on cancel/resume/OTA outcomes.
                                    

                                    

                                        Sub ID
                                        sims.mobility_subscription_id
                                        bulk-activator: after Helix activation, calls subscriber/details for each returned sub ID to confirm the matching ICCID, then writes here. Clicking the Sub ID queries Helix live via the dashboard proxy.
                                    

                                    

                                        Port
                                        sims.port
                                        sms-ingest: updated on every inbound SMS from the port field in the Skyline webhook payload (converted from dot-notation to letter format, e.g. 06.01 → 6A). Also set during fix-sim / retry-activation / manual slot picker.
                                    

                                    

                                        Reseller
                                        reseller_sims.reseller_id
(active = true)		
                                        Written by the dashboard Assign Reseller action (per-row or bulk). A SIM can have multiple historical rows in reseller_sims; only the row with active=true is shown.
                                    

                                    

                                        SMS (24 h)
                                        get_sms_counts_24h() RPC
                                        Aggregated live from the sms_messages table by a Supabase RPC on every page load. Count and last-received timestamp for the last 24 hours. Written by sms-ingest when messages arrive.
                                    

                                    

                                        Last Rotated
                                        sims.last_mdn_rotated_at
                                        mdn-rotator: written after each successful MDN rotation (both cron and manual). Used by the rotator itself to skip SIMs already rotated today.
                                    

                                    

                                        Activated
                                        sims.activated_at
                                        details-finalizer: set once when the SIM transitions to active status. Never overwritten after that.

Bulk Actions (select via checkboxes)

                            
Select multiple SIMs using checkboxes, then use the bulk action buttons. Supported: OTA Refresh, Rotate MDN, Fix SIM, Cancel, Resume, Unassign Reseller, Send SMS, Query.

                            
Shift-click: click one checkbox, then hold Shift and click another — every row between them is set to the state of the second click. Works in the current rendered order (after sort/filter).

                            
Query (bulk action): auto-routes per SIM based on sims.vendor. Helix SIMs query the Helix subscriber details API (by Sub ID); ATOMIC SIMs query ATOMIC subsriberInquiry (by ICCID or MDN). The modal vendor select distinguishes the two — Helix and ATOMIC are separate options.

                        

                        

                            
SIM ID Click

                            
Clicking a SIM's ID opens a modal showing all Helix API log entries for that SIM (looked up by ICCID). Useful for debugging API call history.

                        

                    

                


                
                

                    
Fix SIM — Full Repair Cycle

                    

                        
Fix SIM is the most complex operation. It performs a full repair cycle on a problematic SIM by swapping the IMEI, cycling the carrier status, and refreshing the network profile.

                        

                            
How it works

                            
    
                                
  1. Dashboard sends POST /api/sim-action with action: "fix" to the mdn-rotator worker
    2. 
                                
  2. mdn-rotator resolves the SIM's gateway and port (auto-scans all gateways if not set)
    3. 
                                
  3. Job is queued to FIX_SIM_QUEUE (Cloudflare Queue) to avoid the 30-second service-binding timeout
    4. 
                                
  4. Queue consumer processes the job:
                                    
      
                                        
    1. Gets subscriber details from Helix (MDN + BAN)
      2. 
                                        
    2. Step 1: Cancel — Cancels the subscriber on Helix (reason: CAN, code 1). Retries up to 3 times with retryUntilFulfilled
      3. 
                                        
    3. Waits 10 seconds
      4. 
                                        
    4. Step 2: Resume On Cancel — Resumes the subscriber (reason: BBL, code 20). Retries up to 3 times
      5. 
                                        
    5. Waits 5 seconds
      6. 
                                        
    6. Step 3: IMEI Swap — Retires the old IMEI pool entry, allocates a new IMEI from the pool, checks eligibility with Helix, changes IMEI on Helix, sets IMEI on the physical gateway via SkyLine
      7. 
                                        
    7. Step 4: OTA Refresh — Sends over-the-air refresh via Helix to re-provision the network profile
      8. 
                                    
    
                                
    5. 
                            

                        

                        

                            
If it fails

                            
    
                                
  • Each Helix call uses retryUntilFulfilled which polls the response until the status shows "fulfilled" (up to 3 attempts with delays)
    • 
                                
  • If any step fails, the error is logged to system_errors table and the SIM status is set to error
    • 
                                
  • The queue has max 2 retries — if it fails twice, the message is dead-lettered
    • 
                                
  • Check the Errors page for details. Click the SIM ID to view the full Helix API request/response log
    • 
                            

                        

                    

                


                
                

                    
MDN Rotation (Phone Number Change)

                    

                        
MDN rotation changes the phone number (MDN/CTN) on a SIM. This runs automatically every night and can also be triggered manually.

                        

                            
Nightly Cron (automatic)

                            
    
                                
  1. Cron fires every 20 min from midnight to noon EST (05:00–17:00 UTC) on the mdn-rotator worker
    2. 
                                
  2. All SIMs with status active are queued to MDN_QUEUE
    3. 
                                
  3. Queue processes batches of 10, concurrency 1 (sequential batches)
    4. 
                                
  4. For each SIM:
                                    
      
                                        
    1. Calls Helix MDN change endpoint (PATCH /api/mobility-subscriber/ctn) with the subscription ID
      2. 
                                        
    2. Polls subscriber details until a new phone number appears
      3. 
                                        
    3. Closes old phone number in DB (sets valid_to timestamp)
      4. 
                                        
    4. Inserts new phone number in sim_numbers
      5. 
                                        
    5. Updates last_mdn_rotated_at on the SIM record
      6. 
                                        
    6. Sends number.online webhook to the assigned reseller
      7. 
                                    
    
                                
    5. 
                            

                        

                        

                            
Manual rotation

                            
Click Rotate MDN on a SIM row, or use the bulk action. The same flow runs but only for the selected SIM(s). Manual rotation is vendor-aware: Helix SIMs use the Helix MDN-change flow above; ATOMIC SIMs call swapMSISDN with the SIM's current MDN + a ZIP (defaults to HX_ZIP), then read the new MSISDN back (from the swap response or a follow-up subsriberInquiry). DB writes and number.online webhook are identical regardless of vendor. Note: the nightly cron queue currently only rotates Helix SIMs — ATOMIC SIMs rotate only via the manual action until the queue consumer is ported.

                        

                        

                            
If it fails

                            
    
                                
  • Queue has max 3 retries per message
    • 
                                
  • After all retries fail, error is logged to system_errors
    • 
                                
  • At 07:00 UTC, a separate cron sends an error summary to Slack listing all failed SIMs
    • 
                            

                        

                    

                


                
                

                    
OTA Refresh

                    

                        
OTA (Over-The-Air) refresh sends a network profile update to the SIM via the carrier. This is useful when a SIM has connectivity issues after an IMEI change or MDN rotation.

                        

                            
Steps

                            
    
                                
  1. Dashboard sends action: "ota_refresh" to mdn-rotator
    2. 
                                
  2. Worker gets subscriber details from Helix to obtain the attBan, subscriberNumber (MDN), and iccid
    3. 
                                
  3. Calls Helix OTA endpoint (PATCH /api/mobility-subscriber/ota) with those three fields
    4. 
                                
  4. Helix pushes the updated profile to the SIM over the air
    5. 
                            

                        

                        

                            
Status update

                            
The OTA response includes the current subscriber status. If the status is Suspended or Canceled, the SIM status in DB is automatically updated to match. This means running OTA on a suspended SIM will correctly reflect suspended in the dashboard.

                        

                        

                            
When to use

                            
    
                                
  • SIM not connecting to network after IMEI change
    • 
                                
  • SIM not receiving SMS after MDN rotation
    • 
                                
  • General connectivity troubleshooting
    • 
                            

                        

                        

                            
Failure: data_mismatch

                            
If Helix rejects the OTA request with "The sim number does not match our records", the SIM status is automatically set to data_mismatch. This means the subscriber number (MDN) stored in Helix no longer matches what is on record — typically caused by an out-of-band MDN change. To recover: verify the correct MDN via Helix subscriber details, update the DB record, then retry OTA.

                        

                        

                            
Failure: helix_timeout

                            
If Helix rejects the OTA request with "The requested subscription does not belong to the user.", the SIM status is automatically set to helix_timeout. This means Helix cannot locate the subscription under the current account — typically caused by a subscription being transferred, cancelled on the carrier side, or a credentials mismatch. Investigate the subscription in Helix directly.

                        

                    

                


                
                

                    
Cancel / Resume

                    

                        

                            
Cancel

                            
    
                                
  1. Dashboard sends ICCIDs to the sim-canceller worker
    2. 
                                
  2. For each SIM: gets subscriber details (MDN) from Helix
    3. 
                                
  3. Calls Helix status change: subscriberState: "Cancel", reason code CAN (ID 1)
    4. 
                                
  4. Updates SIM status to canceled in DB
    5. 
                                
  5. Expires current phone number (sets valid_to)
    6. 
                                
  6. Sends sim.cancelled webhook to the assigned reseller
    7. 
                            

                        

                        

                            
Resume

                            
    
                                
  1. Dashboard sends the action to the mdn-rotator worker (action: "resume")
    2. 
                                
  2. Gets subscriber details from Helix
    3. 
                                
  3. Calls Helix status change: subscriberState: "Resume On Cancel", reason code BBL (ID 20)
    4. 
                                
  4. Updates SIM status to active in DB
    5. 
                                
  5. Re-fetches subscriber details and inserts new phone number into sim_numbers
    6. 
                            

                        

                    

                


                
                

                    
Change IMEI

                    

                        
Swaps the IMEI associated with a SIM, both on the carrier side (Helix) and the physical gateway (SkyLine).

                        
    
                            
  1. Dashboard sends action: "change_imei" to mdn-rotator with optional imei (manual) or auto_imei: true (from pool)
    2. 
                            
  2. Retires all existing in_use IMEI pool entries for this SIM
    3. 
                            
  3. If auto: allocates next available IMEI from pool. If manual: uses the provided IMEI
    4. 
                            
  4. Checks IMEI eligibility with Helix (GET /api/mobility-eligibility/imei)
    5. 
                            
  5. Sets IMEI on the physical gateway port via SkyLine API
    6. 
                            
  6. Updates/upserts the IMEI pool entry (gateway, port, sim_id, status=in_use)
    7. 
                            
  7. Changes IMEI on Helix (PATCH /api/mobility-sub-ops/imei-plan) using retryUntilFulfilled
    8. 
                            
  8. Updates the SIM record with the new IMEI and pool entry ID
    9. 
                        

                        

                            
If eligibility fails

                            
The IMEI is marked as retired in the pool and a new one is tried (when using auto). The error is logged.

                        

                    

                


                
                

                    
Retry Activation

                    

                        
Retries a failed SIM activation via the carrier API (ATOMIC for AT&T, Wing IoT for IoT SIMs). The SIM must be in error status.

                        
You will be prompted to choose an IMEI strategy:

                        
    
                            
  • Same IMEI — reuses the IMEI that failed (skips pool retire/allocate). Use when the issue was a transient API error, not an IMEI problem.
    • 
                            
  • New from Pool — retires the old IMEI and allocates a fresh one. Use when the IMEI itself may be the cause of the error.
    • 
                        

                        
    
                            
  1. Worker locates the SIM on a gateway (or uses the slot you provide)
    2. 
                            
  2. Sets the chosen IMEI on the gateway port
    3. 
                            
  3. Calls the carrier activation API (ATOMIC: re-activates and returns MSISDN; Wing IoT: PUT activated status)
    4. 
                            
  4. On success, SIM status is set to active with the new phone number
    5. 
                        

                    

                


                
                

                    
Bulk Activation

                    

                        
Activates multiple new SIM cards at once via the Helix bulk activation API.

                        

                            
Two input modes

                            
    
                                
  • CSV mode (GET /run) — Fetches a Google Sheet CSV with columns: iccid, imei, reseller_id, status. Only rows with status "pending" are processed.
    • 
                                
  • JSON mode (POST /activate) — Receives {sims: [{iccid, imei, reseller_id}]} directly.
    • 
                            

                        

                        

                            
Steps

                            
    
                                
  1. Validates all input rows (must have valid iccid, imei, reseller_id)
    2. 
                                
  2. Skips ICCIDs that already have a mobility_subscription_id (already activated)
    3. 
                                
  3. Gets Helix bearer token
    4. 
                                
  4. Sends all SIMs in one call to Helix bulk endpoint (POST /api/mobility-sub-ops/subscription)
    5. 
                                
  5. ICCID resolution: the bulk response returns only mobilitySubscriptionIds — no ICCIDs, and Helix does not guarantee response order matches request order. The worker immediately calls POST /api/mobility-subscriber/details for all successful sub IDs to get the authoritative ICCID for each from Helix, then writes to DB using those ICCIDs.
    6. 
                                
  6. For each successful activation: upserts SIM record with status provisioning, assigns to reseller
    7. 
                                
  7. For failed items: logs error and sets SIM status to error (failed items do include ICCID in the response)
    8. 
                            

                        

                        

                            
Why ICCID resolution matters

                            
Positional matching (pairing the Nth sub ID to the Nth ICCID we sent) caused SIMs 417–424 to have their ICCIDs shifted by one position after a bulk run. The fix: always confirm ICCID ownership via subscriber_details before writing to DB. Logged as step post_activation_details in Helix API Logs.

                        

                        

                            
After activation

                            
New SIMs start in provisioning status. The details-finalizer worker runs on a cron schedule to poll Helix for the assigned phone number and move the SIM to active.

                        

                    

                


                
                

                    
Send SMS

                    

                        
Sends an outbound SMS from the physical gateway port associated with a SIM.

                        
    
                            
  1. Dashboard resolves the SIM's gateway_id and port from the database
    2. 
                            
  2. Sends the request to skyline-gateway worker at /send-sms
    3. 
                            
  3. Skyline-gateway loads gateway credentials from DB, authenticates with the device
    4. 
                            
  4. Sends the SMS via the Supabase Edge Function bridge (skyline-bridge) to the physical device
    5. 
                            
  5. The gateway transmits the SMS from the specified port's SIM card
    6.

Incoming SMS (sms-ingest)

                    

                        
The sms-ingest worker receives incoming SMS messages from the physical gateways and routes them to the correct reseller.

                        
    
                            
  1. Gateway pushes incoming SMS to sms-ingest via HTTP (two formats supported: JSON recv-sms or raw octet-stream)
    2. 
                            
  2. Worker authenticates the request via header/query/path secret
    3. 
                            
  3. Parses the SMS body (base64 decodes if JSON format)
    4. 
                            
  4. Resolves the SIM by ICCID or phone number lookup
    5. 
                            
  5. Auto-links the SIM to the gateway port — updates port and gateway_id on the SIM using the MAC address in the webhook
    6. 
                            
  6. Generates a deduplication message ID (SHA-256 hash of from+to+body+timestamp)
    7. 
                            
  7. Inserts the message into inbound_sms table
    8. 
                            
  8. Sends webhook to the SIM's assigned reseller (up to 5 retries with exponential backoff)
    9. 
                            
  9. Records delivery result in webhook_deliveries table
    10. 
                        

                        

                            
Port Conflict Resolution

                            
The gateway is the physical source of truth for which SIM occupies a slot. If the incoming SMS reports an ICCID on a port that the DB already attributes to a different SIM, sms-ingest will:

                            
    
                                
  1. Clear port and gateway_id on the old SIM (logged as Evicted SIM X from slot Y/Z)
    2. 
                                
  2. Assign the slot to the SIM whose ICCID just sent the SMS
    3. 
                            

                            
This handles cases where a physical SIM card is replaced in a hardware slot without a corresponding DB update (e.g. after a helix_timeout SIM is swapped out).
